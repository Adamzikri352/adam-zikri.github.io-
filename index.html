I am a software engineering student passionate about building practical, real-world mobile applications using Kotlin and Jetpack Compose. This portfolio documents my journey through a series of labs and two major mobile development projects, where I progressively built SmartTenaga — an Android app for monitoring household electricity usage and promoting energy-conscious habits.

🔌 SDG 7 — Affordable and Clean Energy: SmartTenaga helps Malaysian households track, understand, and reduce their electricity consumption and TNB billing costs.

Labs

A progression of five labs covering UI fundamentals, interactivity, Material Design, navigation with ViewModel, and Room database persistence — each building on the last toward the final SmartTenaga app.

Lab 1 — UI

Static UI with Jetpack Compose

Built a static Wedding Card UI using Jetpack Compose, focusing on layout composables such as Column, Row, and Text to understand the declarative UI model.

Reflection This lab was my first hands-on introduction to Compose's declarative syntax, which felt very different from traditional XML layouts. I learned how layout modifiers like padding and alignment work together to build clean, structured screens. It gave me the foundation I needed before adding any interactivity.
Lab 2 — Interaction

State and User Interaction

Introduced interactivity using TextField and Button composables, managing dynamic UI updates through Compose state with remember and mutableStateOf.

Reflection Understanding recomposition was the biggest takeaway here — seeing the UI update automatically when state changed felt almost magical compared to manual view updates. I also learned the importance of hoisting state correctly so components stay reusable. This lab set the stage for all the form-based screens in SmartTenaga later on.
Lab 3 — Material Design

Theming with Material 3

Applied Material Design principles by customising themes, typography, and colour schemes, and used Card composables to create polished, consistent layouts.

Reflection This lab taught me how powerful a centralised MaterialTheme is for maintaining visual consistency across an entire app. Designing a custom colour palette for SmartTenaga made the app feel much more like a real product rather than a tutorial project. I also appreciated how Card components made information grouping visually clearer.
Lab 4 — Navigation

Multi-Screen Navigation with ViewModel

Implemented Jetpack Navigation Compose to connect multiple screens, with a shared ViewModel managing app-wide state such as appliance lists and login sessions.

Reflection Wiring up NavHost and managing the back stack was challenging at first, especially handling login flow correctly with popUpTo. Sharing a single ViewModel across screens clarified the MVVM pattern for me and made it obvious why state shouldn't live inside individual composables. This lab essentially became the skeleton of the final SmartTenaga app.
Lab 5 — Room

Persistent Storage with Room

Upgraded the app to a full MVVM + Repository + Room architecture, allowing appliance data and bill goals to persist across app restarts using a local SQLite database.

Reflection This was the most technically demanding lab — getting Entities, DAOs, and the Repository layer to work together with StateFlow took real debugging effort, especially around KSP and Gradle version mismatches. Once it worked, seeing data survive an app restart for the first time was incredibly satisfying. It made me appreciate why persistence is non-negotiable for "real" apps.

Projects

Two major milestone projects that combine the lab concepts into a complete, working application aligned with SDG 7 — Affordable and Clean Energy.

Project 1 — Navigation & ViewModel

SmartTenaga

Problem Statement: Many Malaysian households are unaware of how their daily appliance usage translates into their monthly TNB electricity bill, making it hard to set realistic savings goals.

A multi-screen Jetpack Compose app with login, home dashboard, appliance tracking, TNB tiered-rebate bill estimation, usage goals, and a shared ViewModel for app-wide state.

Lessons Learned Building the TNB tiered rebate calculation logic taught me how real-world business rules can get surprisingly complex even for a "simple" formula. I also learned the value of separating UI screens from business logic early, since it made later upgrades to Room in Project 2 far smoother. Overall this project gave me confidence in structuring a multi-screen Compose app from scratch.

Project 2 — API, Sensors & Cloud

SmartTenaga

Problem Statement: Beyond local tracking, users need access to up-to-date energy knowledge, a way to visually document their appliances, and assurance that their data is safely backed up across devices.

Expanded SmartTenaga into a fully connected application with Room persistence, Firebase Firestore cloud sync of appliances and bill goals, live energy articles fetched from the EIA "Today in Energy" RSS feed via Retrofit/OkHttp, and CameraX integration to capture photos of appliances.

Lessons Learned Integrating four different technologies — Room, Firestore, a REST/RSS API, and CameraX — in one cohesive ViewModel really tested my understanding of coroutines, Flow, and StateFlow working together. Handling camera permissions and Firestore sync edge cases taught me to design for failure states, not just the happy path. This project felt like the closest thing to a real production app I've built so far.

Reflection & Learning Journey

Looking back at the full progression from Lab 1 to Project 2.

Across these five labs and two projects, I went from struggling with basic Compose layouts to confidently architecting a multi-screen app with persistent storage, cloud sync, live data, and hardware sensor integration. The biggest shift in my thinking was learning to separate concerns about UI, state management, and data which made each new feature easier to add rather than harder. Debugging Gradle, KSP, and Firebase configuration issues was frustrating at times, but it taught me patience and how to read error logs systematically. I'm proud that SmartTenaga evolved from a static wedding card exercise into a genuinely useful tool aligned with SDG 7, and I feel much more prepared to take on independent Android projects going forward.
